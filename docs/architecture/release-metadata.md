# Release Metadata Contract

## Purpose

Release metadata is the operator-readable and machine-consumable record that connects source, build, artifact, trust, promotion, and runtime deployment.

The MVP does not need a custom release database. It does need a consistent metadata contract so Cloud Build, Cloud Deploy, Binary Authorization, and operators can refer to the same release candidate.

## Contract principles

Release metadata should be:
- **traceable** — every release candidate maps back to source and build execution
- **immutable where required** — artifact identity uses image digest, not only a tag
- **reviewable** — operators can understand why a release is deployable
- **portable across stages** — the same release identity follows dev, stage, and prod
- **small enough for MVP** — no custom catalog or enterprise workflow is required

## Required MVP fields

| Field | Description | Producer | Consumer |
| --- | --- | --- | --- |
| `source_repository` | Repository that produced the release candidate | Cloud Build trigger context | Operators, audit review |
| `commit_sha` | Source revision used for the build | Cloud Build trigger context | Operators, release review |
| `build_id` | Cloud Build execution identifier | Cloud Build | Operators, troubleshooting |
| `build_service_account` | Identity that executed the build | Cloud Build | Trust review, IAM review |
| `image_uri` | Artifact Registry image URI | Cloud Build | Cloud Deploy, operators |
| `image_digest` | Immutable image digest | Cloud Build after push | Binary Authorization, Cloud Deploy |
| `verification_status` | Result of required verification checks | Verification step | Trust decision, operators |
| `verification_timestamp` | Time verification completed | Verification step | Audit review, operators |
| `trust_signal_ref` | Reference to the trust signal or attestation, when available | Verification or attestation step | Binary Authorization, operators |
| `release_name` | Cloud Deploy release name, when available | Cloud Deploy | Operators, promotion review |
| `target_environment` | Environment currently receiving or running the release | Cloud Deploy | Operators, observability |
| `promotion_state` | Current progression status | Cloud Deploy or operator workflow | Operators |

## Optional MVP fields

Optional fields can be added when they are easy to produce without expanding scope:
- `build_log_url`
- `artifact_registry_location`
- `deployed_namespace`
- `deployed_at`
- `runtime_version_label`
- `operator_review_status`

These fields should support review and troubleshooting. They should not introduce a custom release management system in the MVP.

## Metadata lifecycle

### Build time

Cloud Build should create the initial metadata from trigger context, build execution details, and artifact publishing output.

Minimum build-time metadata:
- source repository
- commit SHA
- build ID
- build service account
- image URI
- image digest

### Verification time

Verification should update or emit metadata that records:
- verification status
- verification timestamp
- trust signal reference, when available

A failed verification should still produce enough metadata for operators to understand why the release candidate is not deployable.

### Promotion time

Promotion should preserve the release identity and add target progression context.

Promotion metadata should record:
- release name
- target environment
- promotion state
- deployment result, when available

### Runtime review time

Observability should make release metadata visible enough for operators to answer:
- what was deployed
- where it was deployed
- which source revision produced it
- which image digest is running
- whether the release is healthy enough for further promotion

## Storage expectations

For the MVP, release metadata can start as simple structured output generated during build and deployment steps.

Acceptable early forms:
- JSON generated by Cloud Build
- Cloud Build log output with consistent keys
- Cloud Deploy release annotations or labels where practical
- Kubernetes labels or annotations for runtime correlation

Deferred forms:
- custom database
- release catalog service
- enterprise change management integration
- long-term audit data warehouse

## Validation expectations

The metadata contract should support basic validation before deployment:
- `commit_sha` is present
- `build_id` is present
- `image_digest` is present
- `verification_status` is successful for environments that require it
- `target_environment` matches the intended promotion target

The platform should avoid promoting a release candidate when its metadata is incomplete for the target environment's policy.
